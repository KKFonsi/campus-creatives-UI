==================================================
PHASE 4 PURPOSE
==================================================

Phase 4 focuses on moderation, content safety, featured works, event management, and system administration.

The interfaces should support:

- Reviewing pending student submissions
- Checking ownership and copyright information
- Approving submissions
- Rejecting submissions
- Requesting revisions
- Managing reports
- Selecting featured works
- Managing events and open calls
- Managing colleges and creative categories
- Viewing users and creator accounts
- Viewing platform activity and analytics summaries
- Reviewing official content
- Managing recognition labels

The design must remain:

- Clear
- Fair
- Structured
- Transparent
- Campus-centered
- Accessible
- Consistent with the student experience
- Suitable for a university environment

It must not feel like:

- A generic corporate admin template
- A dark technical control panel
- A social media moderation console
- A police-like enforcement system
- A cluttered spreadsheet
- An unrelated dashboard product

The moderation experience should feel responsible and supportive, not hostile.

==================================================
HCI AND ACCESSIBILITY
==================================================

Follow these principles:

- Keep moderator decisions transparent.
- Clearly show the current submission or report status.
- Show the consequences of actions.
- Use supportive language for revision requests.
- Use rejection only as a clearly final action.
- Keep private moderator notes visually separate from student-facing feedback.
- Clearly distinguish student content from official content.
- Use text and icons alongside status colors.
- Keep filters visible and removable.
- Preserve search and filter context.
- Use breadcrumbs on deep desktop pages.
- Use clear Back behavior on mobile.
- Maintain visible focus states.
- Use accessible table structures.
- Provide labels for all form controls.
- Keep mobile touch targets at least 44px.
- Use confirmation for destructive actions.
- Avoid overcrowding tables.
- Allow tables to transform into cards on mobile.
- Keep analytics readable and not overly decorative.
- Avoid making moderation feel punitive.
- Keep user privacy information appropriately limited.

Desktop target:

- 1440px width
- Maximum content width: 1200–1280px

Tablet target:

- 768px width

Mobile target:

- 390px width
- Side margins around 16px

Desktop:

- Sidebar navigation
- Multi-column dashboards
- Searchable tables
- Side-by-side review layouts
- Persistent moderation controls where useful

Mobile:

- Compact header
- Bottom navigation
- Stacked cards instead of wide tables
- Filter bottom sheets
- Sticky decision actions
- Collapsible detail sections
- One chart per row
- No horizontal overflow unless required for a permission matrix
- Large touch targets

PROTOTYPE CONNECTIONS
==================================================

Connect these flows:

Moderator Review:

Moderator Dashboard → Pending Reviews → Submission Review

Approval:

Submission Review → Approve → Approval Confirmation → Approved State

Approval and Feature:

Submission Review → Approve and Feature → Feature Setup → Success

Revision:

Submission Review → Request Revision → Confirmation → Needs Revision State

Rejection:

Submission Review → Reject Submission → Confirmation → Rejected State

Reports:

Reports Dashboard → Report Detail → Resolve or Escalate → Confirmation

Featured Works:

Moderator Dashboard → Featured Works → Add Feature → Preview → Save

Admin Users:

Admin Dashboard → Users → User Detail → Restrict or Change Role

Admin Events:

Admin Dashboard → Events → Edit Event → Preview → Publish

Colleges:

Admin Dashboard → Colleges → Edit College → Save

Categories:

Admin Dashboard → Categories → Edit Category → Save

Analytics:

Admin Dashboard → Analytics Overview

Roles:

Admin Dashboard → Role and Permission Overview → Edit Role

Switch View:

Moderator or Admin Navigation → Switch to Student View → Student Home

All actions should be front-end-only prototype interactions.

Do not add real persistence.

FINAL REQUIREMENTS
==================================================

Before finishing Phase 4:

- Ensure all screens use the existing design system.
- Ensure the PUP maroon and gold identity remains consistent.
- Ensure desktop and mobile layouts are both complete.
- Ensure moderation language is supportive and transparent.
- Ensure student-facing feedback is separate from internal notes.
- Ensure approval, revision, rejection, and reporting flows are clear.
- Ensure official content is clearly labeled.
- Ensure admin pages do not look like an unrelated generic template.
- Ensure tables adapt into mobile cards.
- Ensure destructive actions use clear confirmations.
- Ensure analytics remain readable and restrained.
- Ensure all actions are interface-only.
- Ensure no backend, database, API, authentication, or persistence is added.
- Do not redesign Phases 1, 2, or 3.
- Do not proceed beyond Phase 4.
